# 关键决策

## 记录规则
当项目出现重要取舍时，在这里记录：
- 决策是什么
- 为什么这么选
- 放弃了什么方案
- 后续如果反悔，应该看什么信号

## 决策列表

### 2026-05-10：公开路线采用“自用工具 + vibe coding 成果展示”
- 决策：Sotto 接下来不急于包装成通用商业产品，而是先作为 Dewens 自用的 Mac 提词工具打磨，同时作为 vibe coding 成果公开到 GitHub；公开表达以 personal macOS teleprompter for vibe coding demos 为主。
- 原因：项目的真实价值来自 Dewens 自己的录屏 / demo 场景和持续迭代过程；公开仓库应展示产品判断、视觉气质、真实试用反馈和代码实现，而不是承诺一个已经成熟的通用提词器。
- 放弃：不立即追求安装包、官网、通用创作者市场、完整 AI 导演能力或商业化包装；公开前先完成真实录屏验收、README 对外化、截图 / 演示素材和仓库卫生检查。
- 反悔信号：如果公开后外部用户的通用提词需求明显强于个人 demo 场景，再考虑把 Roadmap 从自用优先转向通用产品；如果自用录屏仍不稳定，则继续延后公开发布。

### 2026-05-10：最近稿件只是入口，稿件库保存完整历史
- 决策：首页“最近稿件”继续只显示最近 3 篇作为快速入口，但新增“管理全部”进入完整稿件管理页；本地 `PromptDocumentStore` 不再截断为最近 12 篇，而是保存全部稿件。
- 原因：真实使用中“最近稿件”不能替代稿件管理，用户需要一个专门位置管理所有已经准备过的稿件；同时首页粘贴进入准备上场后，稿件应自然成为库里的一个稿件，而不是只存在当前编辑上下文。
- 放弃：不把所有稿件直接堆在首页，也不先做复杂文件夹 / 标签 / 搜索；本轮先形成完整列表、打开、放回首页、删除的基础管理闭环。
- 反悔信号：如果稿件数量增加后完整列表难以浏览，再补搜索、标签、归档和批量管理，而不是继续扩大首页最近区。

### 2026-05-10：将“短语切分”收敛为句子级拆合编辑
- 决策：用户可见的编辑动作改为句子级：选择一句后可以直接改文案、按字间位置拆成两句，也可以与上一句 / 下一句合并；短语仍保留为提词内部节奏与 token 光标推进层，不再作为主要用户心智暴露。
- 原因：真实试用反馈表明“短语切分”容易被理解成“改稿件句子结构”，而此前实现只调整句内短语边界，左侧稿件结构不会变化，导致功能看似无效。
- 放弃：不继续把短语边界作为右侧面板的主操作，也不在本轮引入复杂全文编辑器 / 多选编辑器；先让当前句编辑、拆句、合句和自动保存形成稳定闭环。
- 反悔信号：如果长稿编辑中频繁需要跨多句移动、批量拆合或全文重排，再新增独立全文稿件编辑模式，而不是继续扩大右侧当前句面板。

### 2026-05-09：先落地语音激活播放和屏幕共享隐藏
- 决策：在 Textream 参考能力中优先实现语音激活播放和屏幕共享隐藏；语音激活只用现有麦克风音量门控播放推进，不引入 Speech framework；屏幕共享隐藏通过正式提词窗口 `sharingType` 设置实现，并在设置齿轮中提供开关。
- 原因：这两个能力对真实录屏和自然口播收益最大，且能复用当前 `AudioLevelMonitor`、`TeleprompterSettings` 和 AppKit 提词窗口边界；相比逐字追踪，复杂度和失败面更小。
- 放弃：本轮不做逐字语音识别、不做 token matching、不做远程浏览器同步，也不把阅读模式扩成复杂参数面板。
- 反悔信号：如果语音激活阈值在真实场景中经常误触发或不灵敏，再引入可调阈值或更稳的滑动平均；如果仍然无法解决口播丢位置，再推进 `ReadingToken` + Speech 逐字追踪原型。

### 2026-05-09：Textream 只作为机制参考，不复刻 UI 或直接迁入源码
- 决策：将 `f/textream` 下载到 `artifacts/reference-repos/textream/` 作为外部参考源码，重点吸收逐字语音追踪、语音激活滚动、屏幕共享隐藏、外接屏和远程同步等机制；不复制其 Dynamic Island / Notch UI，也不直接把 Textream 的全局 words 模型迁入 Sotto。
- 原因：Textream 的技术栈和 Sotto 接近，尤其 Speech / AVFoundation / AppKit 窗口层很有参考价值；但 Sotto 的产品气质是暗色点阵、聚光流、像素字体和 AI 演示导演，直接复刻刘海提词器视觉会破坏当前方向。
- 放弃：本轮不实现语音识别、不做浏览器远程控制、不引入多页 `.textream` 文件格式，也不把外部源码纳入 Sotto 主源码提交。
- 反悔信号：如果真实录屏中固定节奏和音量反馈仍不能满足自然口播，优先做语音激活播放；如果仍明显丢位置，再单独推进逐字追踪 spike，并先建立 `ReadingToken` 等提词态附加层。

### 2026-05-07：现场控制改为“少量常驻 + TUNE 抽屉 + 快捷键”
- 决策：正式提词窗口底部默认只常驻上一句、播放 / 暂停、下一句、TUNE 和位置按钮；速度、亮度、字号、透明度、宽度与位置细项收纳进 TUNE 抽屉，同时补充 Prompt 菜单键盘快捷键和全局轻量设置浮层。
- 原因：提词器窗口需要可拖动，底部一排常驻滑杆会和移动窗口、录屏时临场找按钮发生冲突；现场高频操作应足够短，低频微调应可快速呼出但不占据默认阅读焦点。
- 放弃：不把所有现场调节继续常驻在底部，也不做大型设置页或复杂偏好设置窗口；设置齿轮先承接位置、亮度和状态提示，避免把一期做成完整配置中心。
- 反悔信号：如果真实录屏中用户仍然频繁找不到 TUNE，或键盘快捷键比鼠标控制更自然，再把部分调节升为快捷键 / hover 单项；如果 TUNE 抽屉遮挡正文阅读，再改成侧向浮层或靠近当前位置按钮展开。

### 2026-05-07：正式提词进入专注模式，控制项改为紧凑弹性滑杆
- 决策：点击“开始提词”后临时隐藏主编辑窗口，只保留 `Sotto Prompt` 浮窗；提词窗口关闭后恢复编辑窗口。底部现场控制改为紧凑弹性滑杆，覆盖速度、字号、透明度和宽度。
- 原因：真实录屏时编辑工作台会成为视觉噪音，提词态应进入“只看提词器”的专注模式；`- / 数值 / +` 步进器调节效率和质感都偏工具化，ElasticSlider 思路更符合 Sotto 的“表达节奏”动效语言。
- 放弃：不使用 `NSApp.hide` 隐藏整个应用，避免连提词浮窗一起隐藏；不在提词窗口里做大型控制面板或复杂仪表盘，避免抢当前句阅读焦点。
- 反悔信号：如果用户需要边提词边实时回编辑页改稿，再增加“后台”入口或快捷键显隐主窗口；如果滑杆在录屏中仍显拥挤，再把部分控制折叠为 hover 浮层或键盘快捷键。

### 2026-05-07：麦克风声波先做音量反馈，不做自动语音跟稿
- 决策：本轮接入麦克风音量，只用于编辑页预览和正式提词窗口的声波可视化；不根据语音自动推进稿件，也不做语音识别或语义跟读。
- 原因：Dewens 的真实试用反馈需要“声音从中间向两边扩散”的动态生命感，但 MVP 0.1 的稳定闭环仍是确定性切分 + 手动 / 定时播放；自动语音跟稿会引入识别、延迟、权限和失败恢复等额外复杂度。
- 放弃：不在本轮引入 Speech framework、实时转写、AI 跟读或外部音频依赖；不让声波反馈影响提词正文推进。
- 反悔信号：如果真实录屏中手动播放节奏仍明显跟不上口播，且麦克风权限和音频输入稳定，再单独评估“轻量语音跟随”作为 0.2 能力。

### 2026-05-07：提词窗口补桌面级直接操控，仍保持低干扰外观
- 决策：正式提词窗口保留无边框、圆角、透明置顶的 AppKit 面板气质，同时增加隐形四角拖拽缩放命中区、右上角轻量退出按钮、播放态 REC 红色、正文滚动和点击跳句。
- 原因：第一次真实试用暴露出提词窗口已经不只是展示层，还必须支持临场快速调整和纠错；这些属于 Mac 原生提词器的基础操控，不应藏在文字按钮或只能依赖上一句 / 下一句。
- 放弃：不恢复系统标题栏或普通窗口边框；不把控制层做成高亮工具条；不加入复杂位置预设，先把最直接的临场操控补齐。
- 反悔信号：如果隐形四角拖拽仍不够可发现，或无边框 resize 在不同 macOS 版本表现不稳定，再增加更明确的边缘命中区或位置 / 尺寸预设，但避免重新画出破坏圆角观感的可见角标。

### 2026-05-06：先建立轻量 Motion System 边界，不大拆 AppModel
- 决策：本轮只新增 `SottoMotion.swift` 承接 motion token、短字标信号漂移、确认微光、阅读边缘保护和语义图标微动效；`AppModel` 仅修正播放 tick、公告状态和现场控制 clamp，不在本轮拆出完整 PlaybackController / TeleprompterCoordinator。
- 原因：当前 MVP 0.1+ 的主要风险是动效一致性、提词可读性和低动态降级，而不是核心业务模型失控；先把动效组件边界安好，可以减少继续往 `SottoStyle.swift` 堆叠，同时避免大重构打断现有闭环。
- 放弃：不为了架构好看一次性拆分 `AppModel`、`PromptCardLayout` 或 SwiftUI / AppKit 边界；不引入大型动效依赖。
- 反悔信号：如果后续播放控制、窗口控制、键盘控制和录制安全逻辑继续增长，且 `AppModel` 的播放 / 窗口职责明显影响测试和修改速度，再优先抽出薄的 `PlaybackController` 与 `TeleprompterCoordinator`。

### 2026-05-05：MVP 0.1 设计规格确认后纳入动效原则作为验收补充
- 决策：确认 `docs/superpowers/specs/2026-05-02-spotlight-flow-teleprompter-mvp-design.md` 的功能范围不变，同时把 `docs/sotto-interaction-motion-principles-v1.md` 中的顶部聚光、点阵呼吸、状态点呼吸、准备中步骤浮现、切分点反馈和短语级柔光扫读纳入 MVP 0.1 的视觉验收。
- 原因：当前实现已经具备功能闭环，但部分视觉仍停留在静态暗色样式；Dewens 明确指出首页顶部聚光灯等动效文档要求尚未落地，需要逐个补入。
- 放弃：不把动效文档当作 0.2 之后的参考素材搁置，也不只在文档中确认而不进入产品实现。
- 反悔信号：如果真实录屏中持续动效影响阅读或性能，再按场景降低频率、强度或提供更明确的低动态模式，而不是取消「安静呼吸」这个方向。

### 2026-05-05：编辑页改为左长列 + 右侧上下分区
- 决策：稿件编辑页放弃“上编辑、下预览”的整宽上下结构，改为左侧长列独占显示已识别句子，右侧上方放紧凑的短语切分 / 节奏调整面板，右侧下方放提词预览。
- 原因：此前上下结构在固定窗口高度下仍会挤压和重叠；Dewens 明确要求左侧作为完整长列，右侧再分上下两块，并要求右上调整面板缩小到一个面板内完整可见。
- 放弃：不再让提词预览整宽铺在底部，也不再依赖编辑页外层垂直滚动解决空间问题。
- 反悔信号：如果后续句子列表过窄影响长稿扫读，可以增加左列宽度或提供可拖动分栏；如果右上调整项继续增加，再拆成二级面板而不是重新拉高右上区域。

### 2026-05-04：编辑页改为上编辑、下预览工作台
- 决策：稿件编辑页不再继续压缩到竖屏手机镜像比例，也不再显示原稿输入框；改为更宽的横向工作台，上半区左侧展示已识别句子、右侧展示短语切分 / 节奏调整，下半区整宽展示提词预览。
- 原因：全局像素字体后文字占位更大，竖屏下“后台 / 标题 / 面板 / 节奏 / 预览”会互相挤压；Dewens 明确要求左侧只保留已识别句子，并将提词预览作为底部完整区域呈现。
- 放弃：不再用顶部小分段在“节奏”和“预览”之间切换，不再把原稿编辑框放在编辑页主视觉里，也不把所有编辑能力塞进窄竖屏。
- 反悔信号：如果后续需要手机镜像录制版或原稿全文编辑，再单独做一个精简展示模式 / 原稿编辑入口，而不是牺牲主编辑工作台。

### 2026-05-04：所有可见文字统一使用像素字体
- 决策：Sotto 当前版本中所有可见文字内容统一使用内置 Fusion Pixel Font，包括首页、系统提示、状态、控制、稿件输入、用户稿件、编辑内容、预览正文和正式提词正文。
- 原因：Dewens 明确要求“全部文字内容都变成像素风字体”，优先保证参考图的一致视觉气质。
- 放弃：不再保留“提词正文 / 用户输入内容使用普通字体”的例外规则。
- 反悔信号：如果真实提词或长稿编辑时出现明显可读性问题，再考虑只对长正文提供可读模式或字体开关。

### 2026-05-04：主工作台固定一屏，正式提词窗口独立横向放大
- 决策：Sotto 主窗口不再死守手机镜像大小，而是采用稍大的固定工作台，原则是 Mac 上无需纵向下滑即可完成粘贴、编辑、切分和预览；正式提词窗口独立改为横向较大展示，服务真实录屏 / 演示提词。
- 原因：过小的手机镜像尺寸会压缩编辑体验；正式提词场景需要更接近横向悬浮提词器，优先保证可读性和低遮挡。
- 放弃：不再把所有界面统一限制为手机镜像尺寸，也不把正式提词窗口做成窄竖向卡片。
- 反悔信号：如果主窗口在较小屏幕上仍然出现遮挡或底部不可见，再继续压缩单块高度；如果正式提词窗口遮挡演示内容，再增加尺寸预设或位置预设。

### 2026-05-04：像素字体以内置 Fusion Pixel Font 为准
- 决策：Sotto 的像素标题和提词关键文字使用内置 Fusion Pixel Font 简体中文 OTF 资源，运行时注册到进程，不要求用户手动安装系统字体。
- 原因：Dewens 明确要求把控像素风标题字体；内置字体能保证跨机器视觉一致，也符合 SwiftPM App 的可运行交付目标。
- 放弃：不依赖系统已有字体，也不只用 Canvas 点阵模拟作为最终方案；Canvas 点阵保留为质感层和回退形态。
- 反悔信号：如果后续确定更接近参考图的中文像素字体，或 Fusion Pixel Font 在授权、体积、渲染质量上不合适，再替换资源和字体注册入口。

### 2026-05-03：MVP 0.1 采用 SwiftPM + SwiftUI + 少量 AppKit
- 决策：Sotto MVP 0.1 采用 SwiftPM 管理工程，SwiftUI 实现主窗口、编辑页和预览，AppKit 仅用于提词浮窗的置顶、半透明、无边框和窗口层级控制。
- 原因：SwiftPM 能快速搭建可测试、可脚本化运行的 Mac 原生 App；SwiftUI 足够承载当前编辑和预览界面；提词浮窗属于 SwiftUI 不擅长的窗口边界，适合用小范围 AppKit 桥接。
- 放弃：暂不使用 Xcode 项目作为主入口，也不使用 Tauri / Web 原型承载 MVP 0.1。
- 反悔信号：如果后续需要复杂签名、资源管理、App Store 分发或更细的窗口生命周期控制，再评估迁移到 Xcode 项目或引入更完整的 AppKit 架构。

### 2026-05-03：MVP 0.1 的切分先用确定性规则
- 决策：自动切分先使用本地确定性规则，不接 AI、语音识别或外部服务。
- 原因：本阶段核心是验证聚光流提词体验；确定性切分可测试、可离线、稳定，能支撑第一版编辑和预览闭环。
- 放弃：暂不做 AI 可说性检查、素材理解、语义重写和语音跟随。
- 反悔信号：如果真实稿件中大量句子切分不顺、手动调整成本过高，再把 AI 辅助切分或语义节奏识别提前。

### 2026-05-02：第一版定位为 vibe coding 演示提词器
- 决策：第一版产品定位为“面向 vibe coding 和 AI 产品演示的 Mac 原生 AI 提词器”，先服务 Dewens 自己。
- 原因：该场景最贴近实际使用需求，也比通用提词器更有差异化。
- 放弃：暂不先做大众视频创作者提词器、会议发言助手或完整录制剪辑工具。
- 反悔信号：如果后续真实使用中发现通用会议/录制场景需求更强，再调整定位。

### 2026-05-02：第一版核心闭环是材料转口播稿 + 提词体验
- 决策：MVP 优先实现材料输入、AI 生成自然口播稿、演示模式提词。
- 原因：用户核心需求是把想法讲清楚，不只是滚动文字。

### 2026-05-02：设计语言采用暗色精致工作台

### 2026-05-02：MVP 0.1 先做 Mac 本机聚光流提词器
