# 关键决策

## 记录规则
当项目出现重要取舍时，在这里记录：
- 决策是什么
- 为什么这么选
- 放弃了什么方案
- 后续如果反悔，应该看什么信号

## 决策列表

### 2026-05-06：先建立轻量 Motion System 边界，不大拆 AppModel
- 决策：本轮只新增 `SottoMotion.swift` 承接 motion token、短字标信号漂移、确认微光、阅读边缘保护和语义图标微动效；`AppModel` 仅修正播放 tick、公告状态和现场控制 clamp，不在本轮拆出完整 PlaybackController / TeleprompterCoordinator。
- 原因：当前 MVP 0.1+ 的主要风险是动效一致性、提词可读性和低动态降级，而不是核心业务模型失控；先把动效组件边界安好，可以减少继续往 `SottoStyle.swift` 堆叠，同时避免大重构打断现有闭环。
- 放弃：不为了架构好看一次性拆分 `AppModel`、`PromptCardLayout` 或 SwiftUI / AppKit 边界；不引入大型动效依赖。
- 反悔信号：如果后续播放控制、窗口控制、键盘控制和录制安全逻辑继续增长，且 `AppModel` 的播放 / 窗口职责明显影响测试和修改速度，再优先抽出薄的 `PlaybackController` 与 `TeleprompterCoordinator`。

### 2026-05-05：MVP 0.1 设计规格确认后纳入动效原则作为验收补充
- 决策：确认 `docs/superpowers/specs/2026-05-02-spotlight-flow-teleprompter-mvp-design.md` 的功能范围不变，同时把 `docs/sotto-interaction-motion-principles-v1.md` 中的顶部聚光、点阵呼吸、状态点呼吸、准备中步骤浮现、切分点反馈和短语级柔光扫读纳入 MVP 0.1 的视觉验收。
- 原因：当前实现已经具备功能闭环，但部分视觉仍停留在静态暗色样式；Dewens 明确指出首页顶部聚光灯等动效文档要求尚未落地，需要逐个补入。
- 放弃：不把动效文档当作 0.2 之后的参考素材搁置，也不只在文档中确认而不进入产品实现。
- 反悔信号：如果真实录屏中持续动效影响阅读或性能，再按场景降低频率、强度或提供更明确的低动态模式，而不是取消「安静呼吸」这个方向。

### 2026-05-05：编辑页改为左长列 + 右侧上下分区
- 决策：稿件编辑页放弃“上编辑、下预览”的整宽上下结构，改为左侧长列独占显示已识别句子，右侧上方放紧凑的短语切分 / 节奏调整面板，右侧下方放提词预览。
- 原因：此前上下结构在固定窗口高度下仍会挤压和重叠；Dewens 明确要求左侧作为完整长列，右侧再分上下两块，并要求右上调整面板缩小到一个面板内完整可见。
- 放弃：不再让提词预览整宽铺在底部，也不再依赖编辑页外层垂直滚动解决空间问题。
- 反悔信号：如果后续句子列表过窄影响长稿扫读，可以增加左列宽度或提供可拖动分栏；如果右上调整项继续增加，再拆成二级面板而不是重新拉高右上区域。

### 2026-05-04：编辑页改为上编辑、下预览工作台
- 决策：稿件编辑页不再继续压缩到竖屏手机镜像比例，也不再显示原稿输入框；改为更宽的横向工作台，上半区左侧展示已识别句子、右侧展示短语切分 / 节奏调整，下半区整宽展示提词预览。
- 原因：全局像素字体后文字占位更大，竖屏下“后台 / 标题 / 面板 / 节奏 / 预览”会互相挤压；Dewens 明确要求左侧只保留已识别句子，并将提词预览作为底部完整区域呈现。
- 放弃：不再用顶部小分段在“节奏”和“预览”之间切换，不再把原稿编辑框放在编辑页主视觉里，也不把所有编辑能力塞进窄竖屏。
- 反悔信号：如果后续需要手机镜像录制版或原稿全文编辑，再单独做一个精简展示模式 / 原稿编辑入口，而不是牺牲主编辑工作台。

### 2026-05-04：所有可见文字统一使用像素字体
- 决策：Sotto 当前版本中所有可见文字内容统一使用内置 Fusion Pixel Font，包括首页、系统提示、状态、控制、稿件输入、用户稿件、编辑内容、预览正文和正式提词正文。
- 原因：Dewens 明确要求“全部文字内容都变成像素风字体”，优先保证参考图的一致视觉气质。
- 放弃：不再保留“提词正文 / 用户输入内容使用普通字体”的例外规则。
- 反悔信号：如果真实提词或长稿编辑时出现明显可读性问题，再考虑只对长正文提供可读模式或字体开关。

### 2026-05-04：主工作台固定一屏，正式提词窗口独立横向放大
- 决策：Sotto 主窗口不再死守手机镜像大小，而是采用稍大的固定工作台，原则是 Mac 上无需纵向下滑即可完成粘贴、编辑、切分和预览；正式提词窗口独立改为横向较大展示，服务真实录屏 / 演示提词。
- 原因：过小的手机镜像尺寸会压缩编辑体验；正式提词场景需要更接近横向悬浮提词器，优先保证可读性和低遮挡。
- 放弃：不再把所有界面统一限制为手机镜像尺寸，也不把正式提词窗口做成窄竖向卡片。
- 反悔信号：如果主窗口在较小屏幕上仍然出现遮挡或底部不可见，再继续压缩单块高度；如果正式提词窗口遮挡演示内容，再增加尺寸预设或位置预设。

### 2026-05-04：像素字体以内置 Fusion Pixel Font 为准
- 决策：Sotto 的像素标题和提词关键文字使用内置 Fusion Pixel Font 简体中文 OTF 资源，运行时注册到进程，不要求用户手动安装系统字体。
- 原因：Dewens 明确要求把控像素风标题字体；内置字体能保证跨机器视觉一致，也符合 SwiftPM App 的可运行交付目标。
- 放弃：不依赖系统已有字体，也不只用 Canvas 点阵模拟作为最终方案；Canvas 点阵保留为质感层和回退形态。
- 反悔信号：如果后续确定更接近参考图的中文像素字体，或 Fusion Pixel Font 在授权、体积、渲染质量上不合适，再替换资源和字体注册入口。

### 2026-05-03：MVP 0.1 采用 SwiftPM + SwiftUI + 少量 AppKit
- 决策：Sotto MVP 0.1 采用 SwiftPM 管理工程，SwiftUI 实现主窗口、编辑页和预览，AppKit 仅用于提词浮窗的置顶、半透明、无边框和窗口层级控制。
- 原因：SwiftPM 能快速搭建可测试、可脚本化运行的 Mac 原生 App；SwiftUI 足够承载当前编辑和预览界面；提词浮窗属于 SwiftUI 不擅长的窗口边界，适合用小范围 AppKit 桥接。
- 放弃：暂不使用 Xcode 项目作为主入口，也不使用 Tauri / Web 原型承载 MVP 0.1。
- 反悔信号：如果后续需要复杂签名、资源管理、App Store 分发或更细的窗口生命周期控制，再评估迁移到 Xcode 项目或引入更完整的 AppKit 架构。

### 2026-05-03：MVP 0.1 的切分先用确定性规则
- 决策：自动切分先使用本地确定性规则，不接 AI、语音识别或外部服务。
- 原因：本阶段核心是验证聚光流提词体验；确定性切分可测试、可离线、稳定，能支撑第一版编辑和预览闭环。
- 放弃：暂不做 AI 可说性检查、素材理解、语义重写和语音跟随。
- 反悔信号：如果真实稿件中大量句子切分不顺、手动调整成本过高，再把 AI 辅助切分或语义节奏识别提前。

### 2026-05-02：第一版定位为 vibe coding 演示提词器
- 决策：第一版产品定位为“面向 vibe coding 和 AI 产品演示的 Mac 原生 AI 提词器”，先服务 Dewens 自己。
- 原因：该场景最贴近实际使用需求，也比通用提词器更有差异化。
- 放弃：暂不先做大众视频创作者提词器、会议发言助手或完整录制剪辑工具。
- 反悔信号：如果后续真实使用中发现通用会议/录制场景需求更强，再调整定位。

### 2026-05-02：第一版核心闭环是材料转口播稿 + 提词体验
- 决策：MVP 优先实现材料输入、AI 生成自然口播稿、演示模式提词。
- 原因：用户核心需求是把想法讲清楚，不只是滚动文字。
- 放弃：暂不把实时语音跟随、自动读屏、全流程导演作为第一版硬依赖。
- 反悔信号：如果无语音跟随导致真实演示不可用，再提升其优先级。

### 2026-05-02：设计语言采用暗色精致工作台
- 决策：UI 参考 Claudio / AI 电台视频的暗色、微光、状态感和仪式感，但不复刻其品牌或电台设定。
- 原因：Dewens 喜欢该产品的思想与界面设计；它能帮助提词器形成“AI 演示导演”的产品气质。
- 放弃：暂不走纯 Mac 原生白底工具风，也不做强复古像素电台皮肤。
- 反悔信号：如果真实提词时可读性或专业感受影响，再降低视觉风格强度。

### 2026-05-02：MVP 0.1 先做 Mac 本机聚光流提词器
- 决策：第一版实现范围收敛为「已有稿件 -> 自动/手动切分 -> 聚光流提词」，先做 Mac 本机提词窗口。
- 原因：用户明确认为「稿件模式：聚光流提词」是没有它产品就不成立的核心；先验证提词体验，比先做 AI 创作更重要。
- 放弃：MVP 0.1 不做 AI 生成稿件、创作蓝图、要点模式、风格模板、Side Cue、语音识别、Screen Studio 隐藏和第二屏提词。
- 反悔信号：如果 Mac 本机窗口无法有效验证提词体验，或被录屏捕获导致无法使用，再提前推进第二屏或 Ghost Mode 验证。

### 2026-05-02：产品命名采纳 Sotto
- 决策：当前 AI 智能提词器产品命名为 `Sotto`，可写作 `Sotto by Ong`。
- 原因：`Sotto` 有低声、安静、陪伴和表达的气质，适合「录制前的演出后台」与聚光流提词器定位。
- 放弃：暂不使用功能直译类名称，例如 AI Teleprompter、Prompt Tool、Video Script Generator。
- 反悔信号：如果后续发现 `Sotto` 在商标、域名、传播识别或目标用户理解上存在明显障碍，再重新命名。

### 2026-05-02：首页方向采纳演出后台视觉
- 决策：采纳 Dewens 借助 GPT 网页端 DALL-E 3 调整出的首页方向，作为 Sotto 首页 V1 参考。
- 原因：该方向很好地表达了暗色点阵、舞台光、`READY.` 状态、`ON STAGE` 主按钮和「安静后台」气质。
- 放弃：暂不采用普通文本工具、项目管理器或办公软件式首页。
- 反悔信号：如果后续实际交互发现艺术感影响可用性，再在保留气质主线的前提下简化视觉。
